## The one rule that keeps you safe

By default, environment variables are only available on the server. Nothing in `process.env` reaches the browser unless you opt in by prefixing the name with `NEXT_PUBLIC_`. That default is the whole security model: treat every unprefixed variable as a secret, and treat every `NEXT_PUBLIC_` variable as if you printed it on a billboard.

Next.js loads `.env*` files into `process.env` automatically. Keep secrets there, and keep those files out of git.

```txt title=".env"
DB_HOST=localhost
DB_USER=myuser
DB_PASS=mypassword
NEXT_PUBLIC_ANALYTICS_ID=abcdefghijk
```

## Reading secrets on the server

Read server-only values directly in server code: route handlers, Server Components, and the like.

```js title="app/api/route.js"
export async function GET() {
  const db = await myDB.connect({
    host: process.env.DB_HOST,
    username: process.env.DB_USER,
    password: process.env.DB_PASS,
  })
  // ...
}
```

## NEXT_PUBLIC_ is build-time, and it is public

When you prefix a variable with `NEXT_PUBLIC_`, Next.js inlines its value at build time, replacing every `process.env.NEXT_PUBLIC_*` reference with a hard-coded string in the bundle. Two consequences follow.

First: it is public. Anyone can read it in the shipped JavaScript. Never put a secret behind this prefix.

Second: it is frozen at `next build`. The value will not respond to runtime changes, so a single Docker image promoted across environments carries whatever value was set when it was built.

Note that only static references get inlined. Dynamic lookups are skipped:

```js title="app/analytics.js"
// Inlined: direct reference.
setupAnalyticsService(process.env.NEXT_PUBLIC_ANALYTICS_ID)

// NOT inlined: indirected through a variable.
const varName = 'NEXT_PUBLIC_ANALYTICS_ID'
setupAnalyticsService(process.env[varName])
```

## Runtime values without leaking

Need a value resolved at request time instead of build time? Read it on the server during dynamic rendering. Calling `connection()` from `next/server` opts the render into runtime, so `process.env` is evaluated per request.

```tsx title="app/page.tsx"
import { connection } from 'next/server'

export default async function Component() {
  await connection()
  const value = process.env.MY_VALUE
  // ...
}
```

This is what lets one image serve many environments with different values.

## Load order and what to commit

Variables are resolved in order, stopping at the first hit: `process.env`, then `.env.$(NODE_ENV).local`, then `.env.local` (skipped when `NODE_ENV` is `test`), then `.env.$(NODE_ENV)`, then `.env`. The allowed `NODE_ENV` values are `production`, `development`, and `test`.

Commit shared defaults like `.env` and `.env.test`. Never commit `.env*.local` files: those hold your secrets and belong in `.gitignore`.
